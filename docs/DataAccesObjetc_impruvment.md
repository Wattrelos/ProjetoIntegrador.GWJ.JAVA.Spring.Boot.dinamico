💡 Uma dica para o futuro da arquitetura do seu DAO: Eu notei que na sua classe Usuario você já utiliza as anotações do JPA, como @JoinColumn(name = "perfil_id") e @Column(name = "nome_da_coluna").

Se em algum momento você precisar lidar com um banco de dados "legado" onde você não tenha permissão para alterar o nome das colunas, você pode evoluir o seu DataAccessObject.java para que, ao usar Reflection, ele verifique se o atributo possui essas anotações. Se possuir, ele usa o nome especificado na anotação; se não possuir, ele faz o fallback para a convenção padrão (atributo_id). É exatamente assim que o Spring Data JPA/Hibernate funciona por debaixo dos panos!

Para resolver esse problema de forma elegante, podemos criar um método auxiliar no seu DataAccessObject que utilize Reflection para ler as propriedades do atributo (na classe ou em qualquer uma de suas superclasses). Se as anotações @JoinColumn ou @Column estiverem presentes e tiverem a propriedade name preenchida, usamos esse valor. Caso contrário, ele faz o "fallback" e usa a sua função que converte para snake_case.
Com isso centralizado, substituímos as chamadas diretas de convertPascalCaseToSnakeCase(...) por essa nova lógica nos métodos de Inserção, Leitura, Atualização e Filtro.
Com isso, o DataAccessObject passará a identificar se as anotações do JPA (@JoinColumn e @Column) estão declaradas no seu Domínio e extrairá os nomes exatos para formar as consultas do banco, em vez de depender estritamente da conversão literal. O método getFieldInHierarchy permite que, mesmo lidando com o objeto Cliente, ele encontre o campo herdado perfil originado na classe Usuario.
