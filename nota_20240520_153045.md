# Refatoração: Implementação do Padrão Query Builder no DataAccessObject

**Implementação**: 
Extração da lógica de concatenação e geração de strings SQL (como as encontradas na montagem de `INSERT`, geração da cláusula `SET` em `UPDATE`, e nos métodos `buildWhereClause` e `buildM2mSelectSql`) para uma classe (ou conjunto de classes) separada dedicada a ser um Query Builder.

**Motivo**: 
Atualmente, o `DataAccessObject.java` fere o Princípio da Responsabilidade Única (SRP) misturando a manipulação estrutural de texto de banco de dados com a execução e roteamento do JDBC (ex: conexões, transações, prepared statements). O excesso de concatenações manuais dificulta a leitura e manutenção.

**Benefício**: 
A refatoração resulta na diminuição drástica da complexidade da classe DAO. Previne quebra de código por formatação textual incorreta (como esquecer espaços ou vírgulas entre comandos) e, crucialmente, torna viável a escrita de testes unitários apenas para validar a construção de SQL dinâmico de forma independente e sem necessitar de um banco em tempo de teste.

## Questão
Olá. Analisando o **DataAccessObject.java** , que recebe objetos para persistir no banco de dados, um padrão Query Builder seria útil?

## Resposta Completa do Assistente
Olá! Sim, um padrão **Query Builder** seria **extremamente útil** e muito bem-vindo no seu `DataAccessObject.java`.

Analisando o código atual, ele faz um excelente trabalho dinâmico com *Reflection* para extrair campos, valores e mapear nomes de tabelas, inclusive suportando tabelas de herança e recursão. No entanto, ele sofre de um problema arquitetural clássico: **Falta de separação de responsabilidades (SRP - Single Responsibility Principle do SOLID)**. 

No momento, o DAO está fazendo o papel de duas "camadas" diferentes:
1. **Gerador de SQL**: Montando strings de instruções de banco ("na unha", com concatenações, `String.join` e `StringBuilder`).
2. **Executor JDBC**: Gerenciando conexões, transações (`commit`/`rollback`), `PreparedStatement`, configuração de parâmetros e leitura do `ResultSet`.

### Por que o Query Builder seria o encaixe perfeito aqui?

1. **Fim da "Sopa de Letras" (Concatenação)**: Métodos como `buildInsertSql`, `buildWhereClause` e `updateForClass` dependem de lógicas que concatenam aspas graves, espaços e vírgulas. Um erro de digitação de um espaço antes de um `WHERE` quebra o sistema inteiro na hora da execução. Um Query Builder cuida dos espaços e formatações estruturais para você.
2. **APIs Fluentes e Legibilidade**: Ao invés de misturar blocos de `if/else` apenas para saber se você adiciona um `AND` ou `WHERE`, um Query Builder encapsularia isso para que você programasse de forma fluente e expressiva (ex: `builder.insertInto(tabela).columns(colunas).build()`).
3. **Testabilidade**: Hoje, para testar se uma query dinâmica de cliente com filtros está sendo gerada corretamente, você precisa fazer um teste de integração que se conecta ao MariaDB. Com a geração de SQL abstraída em um Query Builder, você poderia criar um Teste Unitário simples que não precisa de banco de dados, apenas afirmando que a saída da string é igual à esperada.
4. **Facilidade de Adaptação (Dialetos)**: Se amanhã o sistema for portado de MariaDB para PostgreSQL, e houver diferenças em sintaxe de paginação ou escape de colunas, bastaria mudar o Query Builder sem precisar tocar na lógica transacional do DAO.

### Como isso se reflete na prática (Exemplo Conceitual)?

```java
String tabela = tablePrefix + convertPascalCaseToSnakeCase(clazz.getSimpleName());
String sql = new SqlQueryBuilder()
                .insertInto(tabela)
                .columns(columns)
                .build();
```
